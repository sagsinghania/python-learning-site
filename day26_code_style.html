Day 26 / Code Style

代码规范不是面子工程，它直接决定别人和未来的你还能不能快速看懂这段程序。

今天你会回到最基础但非常关键的问题：命名要不要清楚、缩进是否统一、注释是否有意义、函数是否太长。

1. 今天目标

今天的目标是建立代码规范意识：写得清楚、写得稳，比堆更多功能更重要。

2. 核心概念

初学阶段最常见的问题不是“不会写”，而是“写出来只有自己当下看得懂”。代码规范训练，就是把这种短视修正过来。

Name

命名要说明用途

像 score_list、save_file 这种命名，比 a、b、doit 强太多。

Shape

结构和格式要统一

缩进层级一致、空行适度、函数不要无限变长，都会提升可读性。

3. 示例代码

下面这类命名和结构，已经比“全是缩写和单字母”强很多。

def show_student_result(name, score): if score >= 60: print(name, "及格") else: print(name, "不及格")
示例在说明什么： 即使功能简单，只要名字清楚、缩进整齐，代码质量就会明显提升。

4. 关键提醒

  • 变量名和函数名尽量直接表达用途。
  • 缩进层级要统一，不要时多时少。
  • 注释写“为什么”，不要写显而易见的“做了什么”。
  • 一个函数太长时，要考虑拆分。
今天最该抓住的点： 零基础阶段养成规范习惯，后面做项目会省掉大量返工成本。

5. 今日练习

P1

重命名变量

找一个旧练习，把模糊变量名改成更清楚的版本。

P2

整理一个长函数

挑一段偏长的代码，想办法拆成两个小函数。

P3

删掉废注释

检查你写过的注释，删掉那种“把代码原样翻译一遍”的低价值内容。

6. 今日总结

  • 代码规范直接影响可读性和可维护性。
  • 清楚命名比炫技更重要。
  • 缩进、空行和函数长度都属于质量的一部分。
  • 规范不是束缚，而是帮你把项目写得更稳。
今天过关标准： 你能把一个旧练习改得命名更清楚、结构更整齐，并能说出修改理由，就说明今天真正收获了代码规范意识。
