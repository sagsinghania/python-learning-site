Day 21 / Debugging

真正能持续写代码的人，不是从不报错的人，而是能稳稳把错误拆开看懂的人。

今天你会开始训练调试能力：怎么看 traceback、怎么判断出错位置、怎么用 print 和分步缩小问题。

最常见的调试起点

先看报错，再看出错行。

numbers = [1, 2, 3] print(numbers[5])

本页目录

今天重点不是新语法，而是面对错误时的思路。

1. 今天目标 2. 核心概念 3. 示例代码 4. 关键提醒 5. 今日练习 6. 今日总结

1. 今天目标

今天的目标是建立最基础的调试流程：先读报错信息，再定位出错行，再判断变量和逻辑哪里不对。

2. 核心概念

很多初学者一见报错就慌，开始随便删改代码。调试能力要反过来：先冷静，把错误当成线索。

Trace

报错信息不是敌人

traceback 会告诉你哪一行出问题、出现了什么类型的错误。

Split

把问题拆小再看

如果一段程序太长，就用 print() 或暂时注释法缩小问题范围。

3. 示例代码

下面这段程序会报错，因为索引超出了列表范围。

numbers = [1, 2, 3] print(numbers[5])
示例在说明什么： 真正的调试不是“把 5 改成别的试试看”，而是先理解：列表只有 3 项，为什么会访问到第 6 个位置？

4. 关键提醒

  • 先看最后一行报错类型，再看上面的出错行。
  • 遇到错误先复现，不要盲改。
  • 可以临时加 print() 看变量当前值。
  • 把问题缩小到最小例子，往往更容易看出原因。
今天最该抓住的点： 调试能力一旦建立，后面学习速度会明显提升，因为你不再每次报错都完全卡死。

5. 今日练习

P1

故意制造一个索引错误

写一个列表，然后故意访问不存在的位置，观察报错内容。

P2

修正输入转换错误

写一个需要 int() 的程序，输入文字触发报错，再加上 try / except 修掉。

P3

用 print 调试

在一个小函数里打印中间变量，观察程序到底在哪一步出现偏差。

6. 今日总结

  • 调试第一步是读懂报错信息。
  • 不要一报错就乱改，先定位问题。
  • print() 仍然是入门阶段很好用的调试工具。
  • 把问题拆小是提高排错效率的关键。
今天过关标准： 你能看懂一个简单 traceback，指出出错行，并用一两步修正逻辑或输入问题，就说明今天的目标达成了。

有了基本调试能力后，就能更稳地完成一段真正像样的小型综合程序。
