Day 25 / Project Structure

项目一旦变长，“能跑”已经不够，目录和文件结构本身也会决定你后面能不能继续维护。

今天你会开始理解小项目应该如何拆分：主入口、功能文件、数据文件、说明文档，各自扮演什么角色。

最小项目结构示意

先看文件角色，不追求复杂目录。

project/ main.py utils.py data.txt README.md

本页目录

今天重点是“怎么摆放代码”，不是新语法。

1. 今天目标 2. 核心概念 3. 示例代码 4. 关键提醒 5. 今日练习 6. 今日总结

1. 今天目标

今天的目标是理解：项目结构不是装饰，而是让你知道入口在哪里、功能在哪里、数据在哪里。

2. 核心概念

很多初学者项目一开始还能看，后面所有功能都塞进一个文件后就会很难继续。项目结构训练就是在问题变大前先建立秩序。

Entry

入口文件要清楚

main.py 或主页面负责组织流程，最好一眼就能知道程序从哪里开始。

Split

功能文件按职责拆分

例如统计函数、文件读写函数、展示函数，不要全部挤在主入口里。

3. 示例代码

下面这个结构已经足够支撑一个零基础阶段的小项目。

main.py # 主流程 helpers.py # 工具函数 storage.py # 文件读写 data.txt # 保存数据
示例在说明什么： 即使项目不大，先把职责拆开，也会让你后面改代码轻松很多。

4. 关键提醒

  • 主入口负责流程，不要什么都写在里面。
  • 工具函数、数据处理、展示逻辑可以逐步分开。
  • 文件名尽量体现用途。
  • 零基础阶段不必追求复杂架构，但要有基本秩序。
今天最该抓住的点： 项目结构的目标不是“看起来专业”，而是让未来的你还能读懂今天写的代码。

5. 今日练习

P1

给一个旧练习拆结构

把你以前的一个单文件练习，想象成主入口 + 函数文件的形式。

P2

写出项目目录草图

为你的待办工具或成绩工具写一个简单目录结构。

P3

标注每个文件职责

在草图旁边写清楚每个文件负责什么。

6. 今日总结

  • 项目结构决定代码是否容易维护。
  • 入口文件、功能文件、数据文件最好职责清楚。
  • 目录和命名本身就是一种沟通方式。
  • 这一步会直接影响后面小项目写得顺不顺。
今天过关标准： 你能为一个小项目写出基本目录结构，并解释每个文件各自负责什么，就说明今天的内容已经掌握。

结构开始清楚后，下一步要把写法本身也变得更干净。
